## Objective

Create three new static SEO landing pages for GitHub Pages:

- `drinking-water-map.html`
- `public-toilet-map.html`
- `free-shower-map.html`

These pages should validate search intent and conversion structure before any interactive map work begins.

## Current Repository Findings

- The site currently uses a dark visual style defined inline in `index.html`.
- Existing design tokens and patterns already exist and should be reused where practical:
  - dark background gradient
  - `Inter, -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Arial, sans-serif`
  - blue accent buttons
  - rounded cards with muted text and bordered containers
  - mobile-first responsive behavior with simple CSS only
- There is no shared CSS file yet.
- There is no `assets/screenshots/` directory content currently present.
- Existing legal/footer links are available and can be reused.
- Existing page contains language switching and analytics logic, but the new step-1 landing pages do not require this functionality.

## Implementation Plan

1. Create one shared stylesheet for the new landing pages.

- Add a small shared CSS file, likely `assets/landing-pages.css`.
- Reuse the current dark palette and component language from `index.html` rather than duplicating large inline style blocks three times.
- Keep CSS lightweight and static, with no external libraries.

2. Build a common page template for all three pages.

- Each page will include:
  - `<!doctype html>` and `lang="en"`
  - `meta viewport`
  - unique `title`
  - unique `meta description`
  - canonical tag pointing to the page URL path
  - optional Open Graph tags if desired, but not required by the prompt
- Body structure will use a single centered container with these sections:
  - hero
  - screenshot preview
  - problem section
  - solution section
  - CTA block
  - SEO content block
  - footer

3. Reuse site styling patterns while simplifying behavior.

- Match existing design system using:
  - same font stack
  - same or very similar CSS variables (`--bg`, `--card`, `--text`, `--muted`, `--accent`, `--line`)
  - button style aligned with `.btn` and `.btn-primary`
  - bordered cards and responsive spacing
- Do not add language switcher, analytics scripts, consent banner, or JavaScript-heavy behavior unless explicitly requested later.

4. Implement screenshot preview with graceful fallback.

- Each page will point to:
  - `/assets/screenshots/water-map.png`
  - `/assets/screenshots/toilet-map.png`
  - `/assets/screenshots/shower-map.png`
- Because these files do not currently exist, the HTML/CSS should degrade cleanly.
- Best implementation approach:
  - include an image element for the expected path
  - wrap it in a styled preview container with overlay text: `Preview of available locations`
  - include a styled fallback layer or placeholder treatment so the section still looks intentional if the image fails or is absent
- Keep fallback implementation static and lightweight.

5. Write page-specific content aligned to search intent.

- `drinking-water-map.html`
  - H1: `Find free drinking water anywhere`
  - SEO title: `Drinking Water Map - Find Free Water Anywhere`
  - description centered on free drinking water, refill stations, taps, and travel convenience
- `public-toilet-map.html`
  - H1: `Find public toilets anywhere`
  - SEO title: `Public Toilet Map - Find Toilets Anywhere`
  - description centered on public restrooms and traveler convenience
- `free-shower-map.html`
  - H1: `Find free showers while traveling`
  - SEO title: `Free Shower Map - Find Showers While Traveling`
  - description centered on shower access during travel, road trips, backpacking, and van life

6. Keep conversion structure identical across pages.

- Hero:
  - exact-match H1 from prompt
  - short problem-focused subheadline
  - CTA button text: `Open full map in app`
- Screenshot preview:
  - large preview area
  - overlay text: `Preview of available locations`
  - small note below: `You are only seeing a limited preview`
- Problem section:
  - short intro and bullet list describing the difficulty of finding these facilities while traveling
- Solution section:
  - explain app utility in generic, non-brand-heavy terms
  - include curated locations, traveler-focused utility, and practical places not shown on most maps
- CTA block:
  - title: `Find more locations nearby`
  - CTA button text: `Open full map in app`
- SEO content block:
  - 150 to 300 words per page
  - natural keyword usage only

7. Link CTA buttons to the current app destinations.

- Reuse existing store links from `index.html` if maintaining current conversion path is preferred.
- Most likely implementation for this step:
  - primary CTA goes to existing app landing destination or store links already used by the site
- If only one CTA target is desired, use the most appropriate existing app destination consistently across all three pages.

8. Reuse footer/legal links.

- Add a simple footer using the current legal pages:
  - privacy policy
  - terms
  - delete data
- Keep footer visually consistent with the main site.
